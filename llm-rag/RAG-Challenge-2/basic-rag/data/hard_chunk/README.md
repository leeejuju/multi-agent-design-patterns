# HardChunk 实验记录与描述

## 什么是 HardChunk？

**HardChunk** 我说一下什么是 HardChunk，因为这是我自己瞎掰的。

它是纯粹根据文本内容，直接从 index 硬切，什么都不加：
1. 不去分析文档的语义结构或布局
2. 也不去分析一些比较重要的原信息（比如 title、company 等）

这些信息统统都不要指的是一种“纯粹数据分块策略

### 核心特征
1. **固定窗口切分**：严格按照预设的字符长度（如 `chunk_size=1500/100`）进行切片。
2. **固定重叠度**：设置固定的重叠区间（如 `chunk_overlap=200/300`），以保留句子的部分上下文。
3. **元数据缺失（Purely）**：在分块内容中，除了原始文本外，不抽取公司名称、文档标题、页面索引等辅助理解的语义信息。
4. **忽视布局结构**：不考虑 PDF 中的表格、标题层级或段落完整性

## 实验结论总结

通过针对5份年报类复杂 PDF 的测试，我们得出以下关于 HardChunk 策略的限制性结论：

### 1. 召回率 (Recall) 尚可，但精准度 (Precision) 低
虽然能够通过关键词召回相关文本，但由于缺乏语义关联，也缺乏额外的理解，召回的内容往往零散，难以支撑高质量推理。

### 2. 语义混淆问题
在没有元数据注入的情况下，模型无法区分特定代称。
* *案例*：当问题提到 "Tradition"（指代 Tradition 公司）时，HardChunk 召回的内容可能只是包含 "tradition" 单词的通用描述，导致模型不对

### 3. 缺乏证据链
即使最终答案正确（Value 正确），其推理过程（Reasoning Process）通常也是脆弱的。模型更多是基于语义模糊匹配（Semantic Match）凑出的答案，而非真正“理解”了文档逻辑。

### 4. 余弦相似度误导
实验发现，有时余弦相似度极高的分块，其实际内容相关性反而不如相似度稍低的分块。这说明简单的向量检索在高复杂度文档面前存在瓶颈。
